## Why

The current Conversions page rollup rail (Month → Week → Source) is single-stage: the user picks one of `pre-discovery | booking | qualified | converted` and every metric in the rail (`Rows | GCLID | Uploaded | Sync | Value`) reflects only that stage. This makes it impossible to compare booked vs. qualified vs. won in a single view, hides whether each conversion went out as GCLID-attributed vs. enhanced-conversion-only (a key Smart-Bidding signal-quality question), and conflates "uploaded to Google Ads" with "accepted by Google Ads" — those are tracked separately on the Upload Report page and the gap is where most Smart-Bidding attribution loss happens. On the per-estimate row the same upload card is reproduced inside a heavy box per row even though the same action is available in bulk from the week/month rollup buttons, adding noise on what should be a scannable list. The expanded detail panel also does not yet link out to HCP or surface customer-identity information (email/phone/address) that field users routinely cross-reference when triaging.

This change re-grounds the rollup rail on metrics that map directly to the Google Ads delivery pipeline (which payload mechanism was used → how many rows were pushed → how many Google actually accepted), adds an `all` mode that surfaces booked / qualified / converted together, and trims the per-row UI in favor of a single inline value, while making the estimate row a launch pad into HCP with richer customer context.

## What Changes

1. **(BREAKING) Add an `all` conversion mode displayed as "All Conv".** A new mode value `all` SHALL be added to `ConversionMode` and SHALL render side-by-side per-stage metrics for booking, qualified, and converted in the same rollup row (month, week, source-group). The mode's display label in `STEP_TABS` SHALL be `All Conv` (short for "All Conversions"). In the tab strip it SHALL be placed in second position, immediately after `Pre-Discovery`, yielding the final order: `Pre-Discovery · All Conv · Booking · Qualified · Converted`. In `all` mode the hierarchy SHALL use **per-stage event dates** for bucketization: each `(estimate, stage)` event is grouped by that stage's own `*_datetime` (`booking_datetime` / `qualified_datetime` / `converted_datetime`). A single estimate that progressed through all three stages therefore contributes one event-row per stage, each bucketed in the period its event actually occurred. The body table in `all` mode follows the same long-format model — one visible row per `(estimate, stage)` event, with a small stage badge — so rollup totals exactly equal the count of visible event-rows in the same window. Rows are NOT grouped by underlying estimate; each `StageEvent` is a fully independent record. Expanding a row opens a detail panel showing only the StageDetail for that row's own stage (not all three stages). Filters and bulk-upload buttons SHALL operate across all three uploadable stages. — *Capabilities: `conversion-view-mode` (modify), `conversions-filtered-totals` (modify), `conversion-pipeline-ui` (modify).*

2. **(BREAKING) Replace the rollup metric rail columns.** The current rail (`Rows | GCLID | Uploaded | Sync | Value`) SHALL be replaced with `Stage | Method | Push | Acceptance | Value`. The same column set SHALL render at the Month, Week, and SourceGroup levels of the hierarchy. In `all` mode each column SHALL show three sub-values — one per stage (booked / qualified / converted) — using a consistent visual layout; in a single-stage mode each column SHALL show one value for the active stage. — *Capability: `rollup-metric-columns` (new).*

3. **Method column — payload mechanism breakdown.** For every conversion candidate row, the page SHALL classify which upload payload was (or would be) used:
   - `with_gclid` — row has a GCLID stored on that stage's `gads_conversion_uploads` row (with or without user data).
   - `user_data_only` — no GCLID, but the estimate's customer has a usable hashed email and/or phone (enhanced-conversion-only path).
   - `none` — neither a GCLID nor a usable user identifier (the row cannot be uploaded).
   The Method column SHALL display the count of rows in each of those three buckets for the active rollup window. Classification SHALL be derived consistently with `supabase/functions/google-ads-conversion-upload/payload-builder.ts`. — *Capability: `rollup-metric-columns` (new).*

4. **Stage column — total per stage.** The Stage column SHALL display the count of conversions actually present at that stage in the rollup window. In `all` mode it SHALL show three counts (booked / qualified / converted); in single-stage mode it SHALL show one. This replaces the existing generic `Rows` cell. — *Capability: `rollup-metric-columns` (new).*

5. **Push column — local upload outcome breakdown.** The Push column SHALL display three sub-counts of `gads_conversion_uploads` rows for the rollup window: `total sent` (status = `uploaded` OR upload was attempted), `sent with no error` (status = `uploaded` AND no error attached), `sent with error` (latest attempt produced an error). Counts SHALL be derived from the existing per-stage status/attempt/error columns already exposed on `vw_conversion_candidates`. — *Capability: `rollup-metric-columns` (new).*

6. **Acceptance column — Google-side acceptance on weekly/monthly rows only.** A new Acceptance metric SHALL be added to Month and Week rollup rows (NOT to SourceGroup rows, NOT to per-estimate rows). The data source SHALL be `vw_gads_upload_reconciliation_daily` — the same view that powers the Upload Report page (`useUploadReport` / `useCombinedUploadReport`). Acceptance SHALL be displayed as `google_successful_count / local_uploaded_count` with a percent. In `all` mode the cell SHALL break down by `event_key` for booked / qualified / converted; in single-stage mode it SHALL show one value scoped to the active stage's `event_key`. — *Capability: `rollup-acceptance-metric` (new).*

7. **Remove the per-row upload block; keep value inline. Collapse all bulk-upload affordances to a single top-of-page button.** On individual estimate rows (`PipelineRowItem`), the right-side upload card containing the `PhaseCell` and the framed Value cell SHALL be removed. The mode-appropriate value SHALL still be displayed on the row but rendered inline without a card container or border — matching the visual treatment of the value cell in the Month / Week / SourceGroup rollup rails. **The per-bucket bulk-upload buttons on Month and Week rollup headers SHALL be removed.** The Conversions page SHALL expose exactly one bulk-upload affordance: the existing Upload button in `ConversionsHeroHeader`, which uploads pending estimates across the currently filtered, visible hierarchy. To restrict the upload scope, users narrow the hierarchy via filters (channel, campaign, assignee, mode) before triggering the top-of-page Upload button. Per-row upload affordance is reachable only through the expanded StageDetail panel for retry workflows. — *Capabilities: `phase-cell-upload` (modify), `conversion-pipeline-ui` (modify), `bulk-upload-scoped` (modify).*

8. **HCP deep-link on the estimate number, and richer customer info in detail.** The estimate label in `PipelineRowItem` (`#{estimate_number}`) SHALL render as an external link to `https://pro.housecallpro.com/app/estimates/{estimate_id}` opening in a new tab. The expanded detail panel SHALL include a new customer-information block surfacing `customer_name`, email, phone, and service address pulled from the existing HCP customer record joined into the conversion-candidates view. — *Capability: `pipeline-row-hcp-link` (new).*

## Capabilities

### New Capabilities
- `rollup-metric-columns`: defines the new five-column rollup rail (`Stage | Method | Push | Acceptance | Value`) and its per-stage layout in `all` mode.
- `rollup-acceptance-metric`: integrates `vw_gads_upload_reconciliation_daily` (Google-side acceptance) into the Month and Week rollup rows of the Conversions page, mirroring the data path used by the Upload Report page.
- `pipeline-row-hcp-link`: HCP deep-link on the estimate label and an expanded customer-identity block (email / phone / address) in the row detail panel.

### Modified Capabilities
- `conversion-view-mode`: adds `all` as a fifth conversion mode and specifies hierarchy/value/upload/detail-panel behavior for that mode.
- `conversions-filtered-totals`: month/week totals are restated in terms of the new Stage / Method / Push / Acceptance / Value columns; the single page-level bulk-upload button is filter-scoped and spans the three uploadable stages in `all` mode.
- `phase-cell-upload`: the per-row PhaseCell rendered in the row's pipeline column is removed; PhaseCell behavior is retained only inside the expanded StageDetail panel for retry workflows.
- `conversion-pipeline-ui`: row layout simplified — no per-row upload card; estimate label is a link; Value renders inline.
- `bulk-upload-scoped`: the per-Week and per-Month bulk-upload buttons are removed; replaced by a single top-of-page Upload button that uploads pending rows across the currently filtered hierarchy (with the same Dialog + countdown-toast flow). All-mode batches span booking + qualified + converted pending IDs and the dialog shows per-stage breakdown.

## Impact

- **Frontend (horizon-dashboard)**:
  - `src/components/conversions/types.ts` — `ConversionMode` gains `'all'`.
  - `src/components/conversions/constants.ts` — `STEP_TABS`, `MODE_CONFIG`, `ROLLUP_METRIC_RAIL_CLASS` (grid template changes for the new 5-column rail with per-stage sub-cells).
  - `src/components/conversions/lib/computeStats.ts` — new return shape: `{ stageCounts, methodCounts, pushCounts, value }` per stage. Replaces flat `RollupStats`.
  - `src/components/conversions/lib/buildHierarchy.ts` — `all` mode groups by `estimate_created_at`.
  - `src/components/conversions/components/hierarchy/MonthCard.tsx`, `WeekBlock.tsx`, `SourceGroupBlock.tsx` — render the new rail; Month/Week additionally render Acceptance.
  - `src/components/conversions/components/primitives/RollupMetricCell.tsx` — supports per-stage tri-cell layout.
  - `src/components/conversions/components/pipeline-row/PipelineRowItem.tsx` — remove the right-side upload card; render value inline; wrap `#{estimate_number}` in an `<a>`; add a customer-information block to the expanded panel.
  - `src/components/conversions/components/pipeline-row/PipelineHeader.tsx` — drop the `Upload | Value` column header since per-row upload is gone; keep Value-only header.
  - `src/components/conversions/hooks/useConversionFilters.ts` — `all` mode does not filter by per-stage status.
  - New hook to fetch the upload-reconciliation rows scoped to the visible hierarchy window (reuses `vw_gads_upload_reconciliation_daily` and the same TZ + week-keying logic from `lib/uploadReport.ts`).
- **Data**:
  - `vw_conversion_candidates` SHALL expose customer `email`, `mobile_number`, and service address fields and a `customer_has_user_identifier` boolean (or equivalent — derivable client-side if email/phone are exposed) for the Method classifier. A migration may be needed if these columns are not already present.
- **Backwards compatibility**: removing the per-row PhaseCell removes an upload entry point that is currently used. The expanded StageDetail panel will be the only per-row upload path; bulk uploads from the rollup are unchanged. `mode === 'qualified'` remains the default to preserve the existing landing experience.
- **Tests**: existing tests under `src/components/conversions/**` referencing `RollupStats`, the per-row upload card, or the four-mode `STEP_TABS` will need updating. New tests should cover (a) `all` mode hierarchy + filtering, (b) Method classifier parity with `payload-builder.ts`, (c) Push column splitting `uploaded` / `error` correctly, (d) Acceptance retrieval from `vw_gads_upload_reconciliation_daily` and per-stage breakdown in `all` mode.
