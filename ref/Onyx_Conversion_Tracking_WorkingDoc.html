⚠ Implementation Audit & Best-Practice Review — added 2026-06-02

This document has been annotated against (a) the live Onyx pipeline — Supabase functions/views queried directly on 2026-06-02, the horizon-dashboard frontend, and the OpenSpec specs — and (b) current Google Ads offline-conversion best practice. All original content is preserved. Every audit note sits in a coloured callout box and carries three layers:

DOC what this working doc claims  ·  BUILT what the code actually does (verified)  ·  BEST what is actually correct — which sometimes sides with the doc, sometimes with the code, and sometimes with neither.

Headline finding: the doc describes a per-option, snapshot-on-first-observation detection model that does not exist anywhere in the system. The live pipeline detects at the estimate level via three SQL functions (get_pending_{booking,qualified,converted}_lead_conversions) and freezes created_at / updated_at once at discovery (ON CONFLICT DO NOTHING). There is no booked_at, no observed_at, no snapshot/event table, no sent/approved-snapshot, and no scheduled+assigned check. Several doc claims are wrong about both the code and the underlying data; a few doc instincts are actually better than the code; and the code carries best-practice gaps the doc never raises.

The whole system in five steps

STEP 1
Catch the events
Detect when an estimate is scheduled, done, and approved/converted.
STEP 2
Freeze the timestamp
Decide the correct status + date for each; snapshot the mutable ones.
STEP 3
Link the lead source
Tie each conversion back to where the lead came from.
STEP 4
Send & confirm
Upload to Google Ads and verify it was accepted, no silent fails.
STEP 5
Monitor for gaps
Flag when a conversion should exist but didn't arrive.

BOOKING_CONFIRMED Detection model not built CRITICAL
Gate — "options[].status = scheduled AND assigned_employees[] non-empty"
DOC fires on dispatch-board state: scheduled + a tech assigned.
BUILT get_pending_booking_lead_conversions() fires whenever any lead-source signal exists on the estimate: is_booking_form = true OR a booking_tags row OR a callrail_leads match. It never reads option status or assignments. So the action really measures "an attributed lead exists," not "booking confirmed." (Schedule & assignment data is imported — into schedules / estimate_assignments — but detection ignores it.)
BEST Pick the semantic deliberately and name the action to match. This action is observation-only, so a bespoke status-snapshot tracker is over-engineering. If "booking" must mean scheduled+assigned, gate on the already-imported schedules/estimate_assignments tables. If it means "attributed lead," rename it and stop calling it BOOKING_CONFIRMED.
Timestamp — booked_at = first-observed scheduled+assigned snapshot; created_at only as a fallback when created_at == updated_at
BUILT timestamp is estimates.created_at, unconditionally — no snapshot, no booked_at/observed_at column exists anywhere in the schema, and there is no created_at==updated_at branch. (99.8% of 4,461 booking rows have conversion_datetime == created_at; the 8 exceptions differ by ~2–3s, a timestamptz-precision artifact.)
BEST For an upper-funnel, non-biddable signal, created_at is actually a fine, immutable anchor — the estimate is born at lead intake, and a few hours' skew is immaterial within a 90-day window. The doc's fear of created_at and its elaborate snapshot machinery are unnecessary here. Keep created_at; delete the snapshot design.
"Still to validate: that updated_at moves when status → scheduled"
BUILT irrelevant to booking — the function never references updated_at. (updated_at is the qualified-stage anchor, not booking.) The honest disclaimer "the how-we-catch-it logic is the proposed concept" is the row's one accurate line — but it's buried while the rest reads as fact.
Qualified leads Gate + anchor chain diverge CRITICAL
Gate — option statuses {complete rated, complete unrated, created job from estimate} OR approved on-site
BUILT gate is estimates.work_status IN ('complete rated','complete unrated') (estimate-level, not per-option) AND at least one estimate_options row with total_amount > 0. "created job from estimate" is not in the gate (it's a separate work_status, ~2,069 rows); there is no "approved on-site" branch. Note: a repo migration (20260526000001) rewrites this gate to approval_status and is recorded as applied, yet the live function still runs the work_status gate — recorded-but-not-effective drift worth resolving.
BEST "Qualified = a real estimate was presented to a visited customer" is best keyed to the visit, i.e. work_status complete + a priced option (close to what's built). The per-option status list in the doc conflates estimates.work_status with the unused estimate_options.status column — drop it.
Anchor — 6-step chain: sent-snapshot → approved-snapshot → completed_at → started_at → on_my_way_at → created_at
BUILT anchor is estimates.updated_at, full stop. The top two "snapshot" anchors do not exist (no status-transition tracker, no sent_at/approved_at; "submitted for signoff" is a current option-status value but its transition time is never captured). completed_at/started_at/on_my_way_at live in work_timestamps and are imported but never read by this function.
BEST The doc's instinct is right and the code is the weakest option here: updated_at is mutable and moves on any re-sync, so it's a poor proxy for "when the estimate was presented." Use work_timestamps.completed_at (immutable, the actual visit end) as the anchor, falling back to started_at, then created_at. Skip the non-existent sent/approved snapshots entirely.
Value — "$250 default"; GCLID — (not described in this row)
BUILT value sent = AVG(all estimate_options.total_amount)/100 — the mean of every priced option (incl. unapproved), not a flat $250 (all 5,056 stored qualified rows carry a real value, avg ~$3,047). $250, if it exists, is a Google-side action default we always override. GCLID = customer_gclids first-touch within 90 days before updated_at (customer-level, not the per-call/per-form match the doc's Step 3 implies).
BEST Qualified is observation-only, so value barely matters — but "AVG of all options incl. unapproved" is a meaningless figure. If you send a value, send the presented value (sum/avg of the options actually quoted). And note: sending a value on a non-biddable action has no optimization effect.
Converted leads Job gate absent; value & anchor differ HIGH
Gate — approved/pro-approved AND a linked Job exists; "confirm a real Job ID (2,102 rows vs 358)"
BUILT gate is only EXISTS an estimate_options row with approval_status IN ('approved','pro approved'). There is no Job requirement — the function never touches the jobs table and returns job_id = NULL.
BEST The doc's "2,102 vs 358" is factually wrong: 3,198 of 3,336 approved estimates already have a linked job (via jobs.original_estimate_id → estimate_option.id → estimate), so a Job gate would drop only ~138, not winnow 6×. A linked Job adds little precision here. If you want the strongest "sale" signal, gate on the job and read its real revenue — but don't justify it with the cited figures.
Anchor — job.created_at (immutable)
BUILT anchor = MAX(approved option.updated_at) — a mutable field that moves on re-sync — even though jobs.created_at is captured in the schema and is unused.
BEST Here the doc is right: an immutable sale time (job.created_at, or the first approval observation) is a better, more stable conversion time than "last time the option row was touched." Switch the anchor to job.created_at with a fallback to the approval time.
Value — "options[].total_amount ÷ 100" (a single option)
BUILT value = SUM(approved options.total_amount)/100 across all approved options (150 estimates have >1 approved option; 149 of them differ from the single-option figure — avg SUM $6,580 vs single $4,563).
BEST Here the code is more correct: a sale's value is total committed revenue, so SUM (or, better, the linked job.total_amount already surfaced in vw_conversion_candidates) beats one option. The doc's singular phrasing understates revenue — fix the doc, keep SUM (or move to job total).
"Confirmed field paths" paragraph LOW
BEST The raw field/ID inventory is accurate (estimate ids csr_*, option ids est_*; schedules.scheduled_start, work_timestamps.*, estimate_options.{status,approval_status,total_amount} all exist). The defect is that the doc wires detection to scheduled_start, work_timestamps.* and option-status — fields the live get_pending_* functions never read. Add a line clarifying which fields detection actually consults (work_status, approval_status, total_amount, created_at/updated_at) vs which are reporting-only, so nobody builds against unused fields. Also: external-export counts ("5,262 option rows", "7,761 estimates") no longer reconcile with the live DB (11,097 option rows) — tag them with source & date.

The 5 sources — directionally right, taxonomy differs MED
How attribution actually resolves (not described in the doc)
BUILT A SQL pre-pass populates customer_gclids from booking-form tags and CallRail (joined by customer_id). booking_lead takes its GCLID per-estimate (booking_tags first, else newest CallRail call). qualified/converted take the oldest customer_gclids entry within 90 days of the conversion time. Channel is resolved entirely in SQL (vw_conversion_candidates) into 7 values — Google Ads / GLS / GMB / Thumbtack / Organic / Direct / Other — and the frontend just trusts it.
BEST Most-correct attribution credits the GCLID from the same interaction that generated this lead (the call/form that created the estimate) — which the system already does for booking. For qualified/converted it falls back to a customer-level click, which can over-credit an unrelated old click. Prefer per-lead attribution; use customer-level only as a fallback.
#1 CallRail [HAVE]   #2 Online booking [PARTIAL]   #3 LSA [GAP]   #4 Referral/organic [GAP]   #5 Recurring [PARTIAL]
BUILT CallRail ✓ (now matched phone→customer, broader than phone→estimate). Online booking — GCLID capture into booking_tags is already live and is the top-priority signal; only GA4/GTM coverage is partial, so the doc understates it. LSA ✓ GAP — detected only via CallRail source strings ("Google Local Services Ads", 612 rows, only 9 with a GCLID → essentially enhanced-conversion-only); no direct LSA API. BEST Keep the channel taxonomy as the single source of truth. Studied edge cases (the "side notes") — current state
BUILT ✓ 90-day click window (the "tightened attribution window", 2026-05-13) is live and matches Google's 90-day lookback. ✓ CallRail-by-customer_id pre-pass (2026-05-22) recovered hundreds of leads whose estimate_id was NULL at trigger time. ✓ "Call forwarding" → Direct (2026-05-25) is live. ✗ Oldest-vs-newest GCLID: resolver picks oldest-in-window (first-touch); the active conversion-attribution-overhaul argues for newest-in-window — not yet built. ✗ "NULL is sticky": once a pending row is inserted with gclid = NULL it is never re-resolved even if a usable GCLID arrives later (NOT EXISTS gate) — a real coverage defect; the re-attribution pass is proposed, not built.

The first-touch vs last-touch choice is a genuine policy decision (last-click is Google's historical default; first-touch better reflects acquisition) — make it explicitly.

Audit — Send to Google Ads & confirm acceptance

Mostly correct — and mostly already built OK
"Store success/error per row. Surface partial-failure errors instead of swallowing them."
BEST ✓ This — the doc's strongest requirement — is fully implemented and is best practice. The uploader sends partialFailure: true, parses partialFailureError per row, and persists per-row error_code/error_namespace/error_detail plus a richer 8-state lifecycle; raw request/response bodies are stored per batch; an automatic pipeline-pause trips on fix-config errors. Step 4/5 are largely "done," not "to build."
Three concrete corrections HIGH
"Missing GCLID = can't upload"
BUILT false — a row with no GCLID uploads via Enhanced Conversions (hashed email/phone); it's skipped (status skipped, lifecycle excluded — a deliberate non-failure) only when there's no GCLID and no contact identifier.
BEST The code is right and the doc is outdated: Google explicitly recommends sending user-provided data even without a GCLID ("conversions with only user-provided data are still useful"), and sending the GCLID in addition to identifiers when present. Rewrite this line.
"conversion_date_time must be in account timezone"
BUILT always emitted in UTC with a literal +00:00 offset, never account time (America/New_York).
BEST An explicit UTC offset is accepted by Google, so it's functionally fine — but Google recommends sending in the account timezone so UI conversion counts and time-of-day reporting reconcile. Switch to account-TZ formatting; it's a one-line fix in hashing.ts.
90-day window — uniform expiry vs the 63-day enhanced-conversion limit HIGH
BUILT pickup.ts expires every queued row whose conversion_datetime is older than 90 days. On the live DB this is the dominant outcome — ~10,309 rows are expired (aged out before upload) vs only ~2,509 sent.
BEST Two issues. (1) Google's window for Enhanced Conversions for Leads is 63 days, not 90 — so GCLID-less rows uploaded at 63–90 days are silently rejected by Google. Expiry should be 63 days for user-data-only rows, 90 for GCLID rows. (2) A 10k-deep expired backlog means the real bottleneck is upload latency / discovery cadence, not the window — fix the throughput (and the stale cron, see Decisions audit) so conversions are sent well inside the window.
Value & data-flow precisions LOW
BEST The "$250 default" belongs to Google's Qualified action, not Converted — and we always override it (we send a computed value for both qualified and converted; booking sends none). Currency is hard-coded USD, so "verify currency" is moot. Data-flow nuance to add: the get_pending_* functions are the discovery step — they freeze rows into gads_conversion_uploads as queued; the upload function drains that queue and never re-runs detection SQL or reads vw_conversion_candidates.

Audit — Best-approach summary & doc quality

What the correct system looks like (independent of this doc and the current code)
BEST 1. Don't build a snapshot/event-sourcing layer. It's the doc's central premise and it's unnecessary. Use the immutable timestamps HCP already gives you, frozen once at discovery.
BEST 2. Anchor each stage to the most meaningful immutable time available: booking → created_at (fine as-is); qualified → work_timestamps.completed_at (visit end), not updated_at; converted → job.created_at (sale), not MAX(option.updated_at).
